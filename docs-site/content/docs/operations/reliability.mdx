## 状态

```bash
mailrelay status
```

输出 Queue 深度、待发送回复、死信数量、最后轮询、运行错误、Catalog Hash 和最近执行。

## 死信重放

修复外部依赖后显式重放，不自动无限重试：

```bash
mailrelay replay queue 42
mailrelay replay reply 17
```

## 72 小时验收

`mailrelay soak` 在受控窗口内跑完整 IMAP / SMTP / Handler 链路并审计结果，用于发布前的可靠性放行。

### 准备

使用一个**专用邮箱**和**一个 Stable HTTP 或 Webhook 命令**。启动前清空 Queue 和 Outbox：

```bash
mailrelay doctor
mailrelay status
```

`doctor` 应当通过、`status` 的 pending / dead 计数应当为 0。

### 启动

```bash
mailrelay soak --duration 72h
```

`soak` 在窗口结束或被信号中断时打印结构化结果，例如：

```text
duration: 72h0m0s
observed: 72h0m3.214s
queue_dead: 0
reply_dead: 0
reply_pending: 0
soak_result: pass
```

### 验收期间

按以下顺序制造事件，让链路经历真实抖动：

1. 发送合法命令若干封，确认审计有成功执行。
2. 发送一封**未授权命令**（错误的 token 或非 allowlist 发件人），确认被拒绝且没有 Handler 被调用。
3. 用**重复的 `Message-ID`** 重复发送同一封命令，确认第二次被去重、没有产生第二次执行。
4. 临时中断 IMAP、SMTP、目标 HTTP 端点，依次恢复，确认 IMAP 重连、Outbox 重发、`mailrelay status` 中错误消失。

### 通过标准

| 检查项 | 通过条件 |
| --- | --- |
| 退出码 | 0 |
| `soak_result` | `pass` |
| 审计 | 0 条未授权执行、0 条重复执行 |
| 结束时状态 | 无 pending / dead Queue，无 pending / dead Reply |

任一项不满足即视为未通过，需修复依赖后**显式**调用 `mailrelay replay` 重放相关死信，再重启验收窗口。

## 备份

停止 MailRelay 后同时备份 YAML 和 SQLite 数据库，或使用 SQLite 在线备份工具。
