# MailRelay v0.1 Reliability Hardening

## Goal

Turn the current feature-complete prototype into a trustworthy single-host alpha. The golden path is authenticated mail to HTTP/Webhook, a threaded reply, and a durable audit trail. The acceptance target is a 72-hour operator-run soak with no duplicate or unauthorized execution and with every failure visible and recoverable.

## Scope

- Persist outbound replies before SMTP delivery and retry them independently from command execution.
- Move exhausted queue jobs and replies into queryable dead-letter state and provide explicit CLI replay.
- Extend `status` with receiver health, pending replies, dead letters, and recent errors.
- Label handlers Stable (`http`, `webhook`), Beta (`workflow`, `queue`), or Experimental (`plugin`, `shell`, `agent`, `mcp`) in generated help and operator docs.
- Add deterministic integration tests around TLS SMTP, IMAP protocol behavior, reconnect/hot reload, SSRF redirect and address checks, and restart recovery.
- Add an opt-in `mailrelay soak` command that runs continuously, emits periodic health summaries, and exits non-zero if acceptance invariants fail. Live mailbox credentials remain operator supplied.
- Add version reporting, GitHub Actions, GoReleaser configuration, checksums, and release notes.

## Data Model

SQLite gains `outbox_replies` and richer queue failure metadata. An outbox row contains message ID, recipient, serialized reply, attempts, next attempt, lease, status, and last error. The message execution transaction stores the audit and outbox record before any SMTP call. Delivery updates only the outbox, so retry never calls a Handler.

Queue and reply state transitions are `pending -> running -> done`; exhausted items become `dead`. Replay is an explicit `dead -> pending` transition that resets lease and scheduling but preserves attempt history in audit data.

## CLI and UX

- `mailrelay status` shows health, last poll, queue depth, pending replies, dead letters, and latest failure.
- `mailrelay replay queue <id>` and `mailrelay replay reply <id>` requeue one dead item.
- `mailrelay soak --duration 72h` runs the normal service plus invariant monitoring and writes a final human-readable report.
- `mailrelay version` prints build version, commit, and build time.
- `doctor` warns when Beta/Experimental handlers are enabled and checks declared executables and outbound host DNS policy.

## Testing and Release

Tests use local TLS/protocol fixtures and injected clocks where possible. CI runs unit tests, race tests, vet, build, and a local golden-path test. Tagged releases build macOS, Linux, and Windows archives with checksums. No live credentials are stored in CI; the soak command is intended for an operator-owned mailbox.

## Non-goals

No Web UI, OAuth, Gmail API, multi-instance synchronization, or new Handler type is added. The work strengthens existing capabilities rather than widening product scope.
