# MailRelay 产品收敛与 Handler 晋级设计

## 目标

把 MailRelay 收敛为可公开发布、适合长期单机自用的 v0.1：它将经过认证的邮件转换为少量、预先声明、可审计且可恢复的远程命令。项目不扩张为通用自动化平台或 Agent 宿主，但会持续加固现有 Handler，使实验能力按证据逐步晋级。

## 产品边界

- 邮件只负责传递 Command 名称、声明参数和认证材料，不能动态选择任意程序、主机、模型、提示词、MCP Server 或未声明工具。
- Command、参数结构、Handler 类型和安全策略全部由本地配置预先声明。
- Router 保持无 Handler 类型分支；所有实现继续使用统一接口和注册表。
- v0.1 不新增 Handler 类型，不增加 Web UI、通用 DAG、任意脚本上传、多租户或多实例协调。
- `http` 与 `webhook` 继续作为公开主路径；其他 Handler 的存在不能增加主路径的配置成本或默认攻击面。

## 核心闭环

公开文档和验收围绕一条完整生命周期组织：

1. 用户安装二进制并运行 `mailrelay init`。
2. 用户只需配置邮箱、发件人白名单、令牌和一个 Stable Command。
3. `mailrelay doctor` 在启动前发现凭据引用、文件权限、DNS/SSRF、可执行文件和成熟度风险。
4. 服务接收邮件，完成解析、认证、去重和参数校验。
5. Handler 在固定策略内执行；执行结果、审计和回复 Outbox 先持久化。
6. SMTP 投递独立重试，不重复执行 Handler。
7. `mailrelay status` 显示接收、执行、Queue、Outbox、死信和最近安全分类失败。
8. 运维者修复依赖后，通过显式 replay 恢复死信任务或回复。

任何功能若不能进入这条可观察、可恢复的生命周期，就不能宣称稳定。

## 成熟度模型

Handler 使用 `Experimental -> Beta -> Stable` 单向晋级。标签由实际保障决定，不由功能完成度或使用时间决定。

### Experimental

- 必须显式启用。
- 允许协议和配置发生不兼容调整。
- 文档必须列出已知风险和不适合的场景。
- 不出现在默认配置、五分钟教程和主路径示例中。

### Beta

- 配置格式和执行语义已基本固定。
- 具备单元、集成、失败恢复和敏感信息回归测试。
- 所有外部副作用都有超时、大小或次数上限。
- `doctor`、`status` 和审计能够解释主要失败。
- 至少完成一次本地故障演练并记录剩余限制。

### Stable

- 满足 Beta 全部条件。
- 重启、重复投递、超时和依赖中断不会导致未记录的重复副作用。
- 默认策略安全；危险能力需要显式配置。
- 错误、日志和 SQLite 不持久化令牌、密码、敏感参数或原始供应商响应。
- 有面向运维者的配置、恢复和兼容性承诺。
- 全量测试、race、vet、跨平台构建和短时 soak 均通过。

## 分阶段晋级

### 阶段一：Workflow 与 Queue

目标是将二者从 Beta 推进到 Stable。

- Workflow：验证步骤上限、总超时、参数映射、直接和间接递归、部分失败语义、审计关联及重启后的可解释性。
- Queue：验证持久化幂等键、事务租约、租约过期恢复、最大尝试次数、指数退避、死信 replay、敏感参数联合脱敏及目标 Command 删除后的行为。
- 两者必须证明不会绕过目标 Command 的参数定义、安全策略或成熟度提示。

### 阶段二：Plugin 与 Shell

目标是先从 Experimental 晋级 Beta，再依据跨平台与资源隔离证据决定是否晋级 Stable。

- 只允许配置中的绝对可执行路径；邮件不能影响 executable、环境变量名或工作目录。
- 禁止 shell 字符串拼接；参数逐项传递。
- 限制执行时长、stdout/stderr 大小、请求大小和继承环境。
- Plugin 的 stdin/stdout JSON 协议必须版本化并严格校验。
- 覆盖进程退出、挂起、超量输出、畸形结果、取消和 Windows/macOS/Linux 差异。

### 阶段三：Agent 与 MCP

目标是从 Experimental 晋级 Beta；只有在远程依赖和非确定性风险得到充分约束后再评估 Stable。

- Agent 的 endpoint、model、system prompt、API key 和预算限制必须由配置固定；邮件只能提供已声明参数。
- Agent 输出必须经过大小、结构和错误分类校验，原始供应商响应不得进入审计。
- MCP Server、transport 和 tool allowlist 必须固定；邮件不能选择 Server 或未列出的工具。
- HTTP transport 复用统一 SSRF、DNS 重绑定、重定向和响应大小策略；stdio 复用 Plugin 进程边界。
- 增加调用次数、超时、响应大小和可选成本上限；明确非确定性结果不等于执行成功。

## 可用性收敛

- 默认生成的配置只展示 Stable 主路径，高风险示例放入独立文档。
- `doctor` 给出可操作的错误和成熟度警告，不只返回配置解析失败。
- CLI 输出保持稳定、可脚本解析，并为人类提供下一步恢复动作。
- 文档中的配置键、默认值、命令、成熟度和实际代码必须由自动检查保持一致。
- GitHub Pages 在桌面与移动断点均验证导航、正文、代码块和搜索可用性；正文阅读宽度不限制整个 DocsLayout。

## 稳定性验证

每个阶段使用同一套证据：

- TDD：先添加能复现缺口的失败测试，再做最小修复。
- 包级测试与 `go test ./...`。
- `go test -race ./...` 和 `go vet ./...`。
- Linux、macOS、Windows 构建检查。
- 本地 IMAP、SMTP、HTTP、进程和 MCP 协议夹具，不依赖真实凭据。
- 短时故障演练覆盖断网、超时、重启、重复 Message-ID、租约过期和死信 replay。
- 发布前 72 小时真实邮箱 soak 仍是运营验收，不作为日常开发循环的阻塞项。

## 发布判定

v0.1 可以发布，当且仅当：

- Stable HTTP/Webhook 主路径的首次安装、执行、回复和恢复闭环可复现。
- Workflow/Queue 达到 Stable；其余 Handler 保持真实、保守的成熟度标签。
- 默认配置与主教程不暴露 Experimental 能力。
- 自动化验证全部通过，且没有未分类的高风险安全或数据丢失问题。
- 已知限制、备份、升级、故障恢复和 Handler 成熟度对用户透明。

实验 Handler 未全部晋级 Stable 不阻塞 v0.1；虚假的 Stable 标签会阻塞发布。
